ocker engine package is characterized by a client server architecture. The Docker server contains the Docker daemon,a background process responsible for creating and managing Docker objects like images, containers, volumes, and networks. The machine where the Docker server runs is called the Docker host. Each machine where you want to run containers should be a Docker host, so it should have a Docker daemon running.

The Docker daemon exposes an API you can use to send instructions, such as to run a container or create a volume. The Docker client talks to the daemon through that API. The client is command-line based and can be used to interact with the Docker daemon either through scripting (for example, Docker Compose) or through the Docker CLI directly.

The Docker CLI is the client. It is used to interact with the Docker server. The Docker server is managing all resources ( e.g, images, containers and networks ) through the Docker daemon. The server can also interact with container registries to upload and download images.

Basic terms

  1. Images: Container Images are lightweight executable packages that include everything needed to run the application that’s inside.

Let's understand how to create images

Container images are the product of executing an ordered sequence of instructions, each resulting in a layer. Each image is made up of several layers, and each layer represents a modification produced by the corresponding instruction. The final artifact, an image, can be run as a container.

Images can be created from scratch or starting from a base image. For example, you can start from an Ubuntu image and apply a series of modifications on top of it. The sequence of instructions would be as follows:
  1. Use ubuntu as base image
  2. Install jre
  3. Run java --version command

All layers in a container image are read-only. Once they are applied, you can’t modify them anymore. If you need to change something, you can do so by applying a new layer on top of it (by executing a new instruction). Changes applied to the upper layers will not affect the lower ones.

When an image is run as a container, one last layer is automatically applied on top of all the existing ones: the container layer. It is the only writable layer, and it’s used to store data created during the execution of the container itself. At runtime, this layer might be used to generate files required by the application to run or maybe to store temporary data. Even though it’s writable, remember that it’s volatile: once you delete your container, everything stored in that layer is gone.

Dockerfile for above set of instructions.

FROM ubuntu:22.04
RUN apt-get update && apt-get install -y default-jre
ENTRYPOINT ["java", "--version"]
  1. Bases the new image on the official image for Ubuntu, version 22.04. By default, Docker is configured to use Docker Hub to find and download images. That’s where the ubuntu:22.04 image comes from.
  2. Installs the JRE using familiar bash commands
  3. Defines the execution entry point for the running container. The java --version command is the entry point of the executing container. If you don’t specify any entry point, the container will not run as an executable.
  4. Once you have the specifications for creating a container image declared in a Dockerfile, you can use the docker build command to run all the instructions one by one, producing a new layer for each of them.

List of some basic docker commands to verify images and run images as a container

  1. docker images // list all docker images
  2. docker run my-custom-image:1.0.0 // run image my-custom-image:1.0.0 as a container
  3. docker ps // list all the containers

Suppose you are given a springboot web application using mysql relational database for storing data.

Springboot lets you package your application as standalone JARs, including everything they need to run except the JRE.

  1. First, you need to identify which image you want to base yours on. You could choose an Ubuntu image, as we did before, and then explicitly install the JRE, or you could choose a base image that provides a JRE already (e.g, Eclipse Temurin 17), which is more convenient. All the major OpenJDK distributions have a related image available on Docker Hub.
  2. Then you need to copy the JAR file of catalog service into image itself
  3. Finally, declare that the entry point for the container execution is the command to run the application on the JRE.

Dockerfile

  1. FROM eclipse-temurin:17
  2. WORKDIR onelap
  3. ARG JAR_FILE=build/libs/*.jar
  4. COPY ${JAR_FILE} catalog-service.jar
  5. ENTRYPOINT ['java', '-jar', 'catalog-service.jar']

Explain Steps in Dockerfile

  1. Ubuntu base image with the Eclipse Temurin distribution of the JRE pre-installed
  2. Changes the current working directory to "onelap"
  3. Builds argument specifying the location of the application JAR file in your project
  4. Copies the application jar file from the local machine into the image
  5. Sets the container entry point to run the application

This Dockerfile declares a JAR_FILE argument that can be specified when creating the image with the docker build command.

Let's build the JAR artifact for the Catalog Service application using the command.
./gradlew clean bootJar

Let's build the container image by running this command:
docker build -t catalog-service .

when you ran Catalog Service as a container, you mapped port 8080, on which the application exposed its services, to port 8080 on your local machine. After doing that, you could use the application by visiting http://localhost:8080. What you did there is called port forwarding or port mapping or port publishing, and it’s used to make your containerized application accessible from the outside world.

By default, containers join an isolated network inside the Docker host. If you want to access any container from your local network, you must explicitly configure the port mapping. For example, when you ran the Catalog Service application, you specified the mapping as an argument to the docker run command: -p 8080:8080 (where the first is the external port and the second is the container port).

  1. docker network create catalog-network
  2. docker network ls
  3. docker run -it --rm -p 3306:3306 -e MYSQL_ROOT_PASSWORD=password -e MYSQL_DATABASE=catalogdb -v ${PWD}/db:/var/lib/mysql --net catalog-network --name catalog-db mysql:8
  4. docker run -d --name catalog-service --net catalog-network -p 8080:8080
